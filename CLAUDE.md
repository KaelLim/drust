## Architecture at a glance

- `meta.sqlite` (management plane): admins, sessions, tenants, hashed bearer tokens. Admin password rotation: `src/bin/set_admin_password.rs` CLI, reads the new password from stdin and hashes via argon2id.
- `tenants/<id>/data.sqlite` (data plane): one SQLite per tenant.
- Reads route through `SQLITE_OPEN_READONLY` connections with `sqlite3_set_authorizer` whitelist (see `src/query/authorizer.rs`). Cross-tenant `ATTACH` is denied.
- **Per-collection DML capability** (v1.6.0+): every collection's schema metadata declares `anon_caps`, a subset of `{select, insert, update, delete}`. Default `[select]` (status quo). Service is unrestricted. The check fires at REST handler entry against a per-tenant `SchemaCache` (`src/storage/schema_cache.rs`); cache invalidation lives in the DDL paths and the admin UI's anon_caps endpoint. Persisted in the per-tenant `_system_collection_meta` table (one row per collection). **Scope: `anon_caps` governs `/records/*` only, NOT `/query`.** The free-form `POST /t/<tid>/query` endpoint accepts arbitrary SELECT from any non-system table by any token (anon or service); parsing every SELECT to evaluate per-table caps is intentionally not done. To fully restrict anon read access on a collection, revoke the anon token rather than relying on `anon_caps:[]`. The admin UI's anon-caps editor surfaces this caveat inline. The `_system_*` tables are blocked from `/records/*` for both anon and service (404), independent of the cap setting.
- **Stored RPCs** (v1.6.0+): tenant-local `_system_rpc` table holds named SELECT functions, callable via REST `POST /drust/t/<id>/rpc/<name>` and the MCP `call_rpc` tool. Service-only authoring; per-RPC `anon_callable` flag gates the REST anon path (MCP is service-only unconditionally). Counters (`anon_calls`, `service_calls`, `last_called_at`) bumped through the writer mutex regardless of caller. SQL is validated at create time via `prepare()` under the read-only authorizer. **Test playground** (v1.7.2+): `/admin/tenants/{id}/_rpc/{name}/test` renders type-aware param inputs, runs against a read connection, and shows result rows + `duration_ms` + `EXPLAIN QUERY PLAN`. Reuses `execute_read_query_with_named` + `validate_and_bind` directly — no duplication of the gating logic.
- Writes go through structured REST/MCP tools against a serialized writer mutex; schema enforcement at tool layer. `FieldSpec` supports allowlisted SQL defaults (`{"sql": "datetime('now')"}`) and foreign keys (`foreign_key: "<target>"` emits `ON DELETE RESTRICT`).
- **Per-collection indexes** (v1.8.0+): tenants can `CREATE INDEX` / `DROP INDEX` via MCP tools (`create_index`, `drop_index`), REST (`POST/DELETE /t/<id>/collections/<coll>/indexes`), or the admin UI's per-collection page (Indexes section). Composite + unique supported, auto-named `idx_<coll>_<f1>_<f2>...`. Large-table guard `DRUST_INDEX_LARGE_TABLE_ROWS` (default 1M) returns 409 LARGE_TABLE unless `force: true`. EXPLAIN QUERY PLAN exposed via `POST /t/<id>/query/explain` (anon-allowed, same authorizer as `/query`) and as a textarea on the admin collection page. Audit-log entries for index DDL include `index_name`, `index_fields`, `row_count`, `force_used`.
- **Per-tenant OAuth login** (v1.12.0+): each tenant registers its own Google/GitHub OAuth apps for its end users (`_system_users`); mirrors Supabase **Project Auth providers**. Config lives in the per-tenant `_system_oauth_providers` table (idempotent migration; fresh tenants get it via `SCHEMA_SQL`) — columns: `provider`, `client_id`, `client_secret` (plaintext at rest, mode-600 SQLite file), `allowed_redirect_uris` (comma-separated frontend URLs, exact-match validation). Three equivalent config surfaces: admin UI virtual sidebar entry `🔐 _oauth_providers` at `/drust/admin/tenants/<id>/_oauth_providers`, REST `GET/PUT/DELETE /drust/t/<id>/admin/oauth-providers[/{provider}]` (service-key only; secrets always redacted on GET), and 3 MCP tools `set_oauth_provider` / `list_oauth_providers` / `delete_oauth_provider` (service-key only). End-user flow: frontend hits `GET /drust/t/<tid>/oauth/<provider>/start?redirect_uri=<frontend_cb>` → drust 302s to provider → 10-step callback at `/drust/t/<tid>/oauth/<provider>/callback` (provider lookup → state CSRF via `subtle::ct_eq` → PKCE verifier from cookie → TOCTOU-safe redirect_uri allowlist re-check → token exchange → `email_verified` flag → `allow_self_register` gate / auto-link by email (existing row wins, profile untouched) / auto-create with sentinel `password_hash="$oauth-only$"` → user-session create → audit) → 302 to frontend `<cb>#access_token=drust_user_xxx&token_type=Bearer&expires_in=2592000` (Supabase/Auth0 URL-fragment pattern). Password login on the sentinel hash returns `401 INVALID_CREDENTIALS` (same-timing dummy verify); `/me/password` returns `409 OAUTH_ONLY_NO_PASSWORD`. Per-IP rate-limit 5/min on `/callback` only (XFF[-2], reuses v1.9 `IpRateLimit`); `/start` is cheap, unthrottled. Audit rows enriched with `auth_method=oauth_<provider>`, `oauth_email`, `oauth_error_code`, `auth_user_id` on both success and failure paths; any callback failure 302s to `<frontend_cb>#error=<code>` (codes: `oauth_misconfigured`, `oauth_state_mismatch`, `oauth_invalid_redirect`, `oauth_provider_error`, `oauth_email_unverified`, `oauth_not_allowed`). `src/oauth/` library (`OauthProvider` trait + Google/GitHub adapters + state/PKCE helpers) is the SAME library v1.11 admin OAuth uses — `TenantAuthState` gained `oauth_adapter_override: Arc<HashMap<String, Arc<dyn OauthProvider>>>` for test-only injection (empty in production). 17 integration tests in `tests/tenant_oauth.rs` cover both happy paths (Google + GitHub), 4 negative categories (state/PKCE, invalid_redirect at-start + TOCTOU at callback, provider error, email_unverified, not_allowed / auto_create / auto_link, sentinel-blocks-password), cross-tenant isolation (config + users), and audit-row enrichment. Spec: [`docs/superpowers/specs/2026-05-15-drust-per-tenant-oauth-design.md`](../docs/superpowers/specs/2026-05-15-drust-per-tenant-oauth-design.md). Setup: [`docs/oauth-setup.md`](docs/oauth-setup.md#per-tenant-oauth-v112).
- **Admin OAuth login** (v1.11.0+): the `/drust/login` page gains Google and GitHub OAuth buttons next to the existing username + password form. Config via env — `DRUST_OAUTH_GOOGLE_CLIENT_ID/SECRET`, `DRUST_OAUTH_GITHUB_CLIENT_ID/SECRET`, `DRUST_ADMIN_OAUTH_ALLOWED_EMAILS`, `DRUST_PUBLIC_URL`. Buttons render only when both `CLIENT_ID` and `CLIENT_SECRET` of a provider are set (partial config logs a `warn` and skips). `admins.email` nullable column added via idempotent migration; partial unique index `idx_admins_email` enforces uniqueness when present. Populate via `set_admin_password --username admin --email you@example.com`. Callback executes a 9-step chain in `src/mgmt/oauth_login.rs::oauth_callback`: provider lookup → state CSRF (constant-time via `subtle::ct_eq`) → PKCE verifier from cookie → token exchange → `email_verified` flag → email allowlist (lowercased on parse + match) → admin row lookup (`COLLATE NOCASE`) → session create → audit; any failure 302s to `/drust/login?oauth_error=<code>`. Google `id_token` is base64-decoded WITHOUT signature verification per OIDC Core §3.1.3.7 (confidential client; channel trust via TLS to token endpoint). GitHub flow does 3 round-trips: `/access_token` → `/user/emails` (picks primary+verified) → `/user`. Sessions reuse the existing `sessions` table — same shape as password login. Audit rows enriched with `auth_method` / `oauth_email` / `oauth_error_code` for both success and failure paths; `admin_id` attached on success via `with_extra`; `tenant` / `token_hint` are `"-"` for admin-plane rows. `src/oauth/` is an actor-agnostic library (`OauthProvider` trait + Google/GitHub adapters + `ProviderRegistry` driven from env) — v1.12 per-tenant OAuth will plug into the same trait. 12 integration tests in `tests/admin_oauth.rs` cover the foundation (fake provider HTTP server) + 2 happy paths + 9 negative paths (state mismatch, PKCE/provider error, email unverified, not_allowed, admin_email_missing, button hidden, audit logged, password regression). Spec: [`docs/superpowers/specs/2026-05-14-drust-admin-oauth-design.md`](../docs/superpowers/specs/2026-05-14-drust-admin-oauth-design.md). Setup: [`docs/oauth-setup.md`](docs/oauth-setup.md).
- **Vector storage & similarity search** (v1.10.0+): `FieldSpec` accepts a new `vector` type with required `dim` (1..=4096); lowers to a SQLite BLOB column holding packed little-endian f32 values. Per-collection vector fields are persisted in `_system_collection_meta.vector_fields_json`. Records CRUD packs JSON arrays to BLOB on INSERT/UPDATE (422 on dim mismatch / NaN / Inf — `VECTOR_DIM_MISMATCH` / `VECTOR_NON_FINITE`); GET and list **exclude** vector columns from responses (default-hide, no opt-in mechanism in v1). Similarity search endpoint `POST /t/<id>/collections/<c>/search` takes a structured body with `field`, `vector`, `k` (1..=1000), `metric` (cosine | l2 | l1), `where` (Filter AST — no raw SQL), and `select`. drust constructs the SQL itself; anon callers need `select` in anon_caps; user tokens with read_scope=own get an auto-appended `<owner_field> = :user_id` clause; service bypasses. MCP `search_collection` tool mirrors the REST shape 1:1. sqlite-vec is registered as a SQLite auto-extension once at process startup (OnceLock-gated, called from both `open_write` and `open_read` before `Connection::open*`), so `vec_distance_cosine/l2/l1` are also callable from `/query` (service token) and stored RPCs. v1 ships brute-force scan only; ~10⁵ rows is the scale ceiling on a modest CPU before considering Approach B (vec0 paired index — non-breaking upgrade, deferred). Spec: [`docs/superpowers/specs/2026-05-13-drust-vector-search-design.md`](../docs/superpowers/specs/2026-05-13-drust-vector-search-design.md).
- **Per-tenant end-user auth** (v1.9.0+): per-tenant `_system_users` + `_system_sessions` tables hold registered users and opaque session tokens (SHA-256-hashed, sliding 30d expiry, prefix `drust_user_`). Three-kind bearer resolution in `bearer_auth_layer`: user-session lookup first (per-tenant), falls through to existing service/anon meta lookup; attaches `AuthCtx` (Anon/Service/User { user_id, token_hash }) for downstream handlers. REST endpoints: `POST /t/<id>/auth/{register,login,logout,logout-all}`, `GET/PATCH /t/<id>/me`, `POST /t/<id>/me/password`. Self-register is per-tenant opt-in (`tenants.allow_self_register`, default 0; admin UI checkbox on `_api_keys` page). Brute-force defenses: per-IP rate-limit on login (5/min) and register (3/min), IP extracted as XFF[-2] (S3); login uses argon2id with a fixed `DUMMY_HASH` to equalize timing across known/unknown email paths (S1). Per-collection row-level filter via `owner_field` + `read_scope` in `_system_collection_meta`: when set, user tokens see only own rows (read_scope=own) or everyone's (all); INSERT overwrites owner_field with caller's user_id; UPDATE/DELETE foreign rows return 404 (no enumeration); anon is denied on owner-scoped collections; service bypasses but must populate owner_field on INSERT (409 OWNER_FIELD_REQUIRED). User tokens fall through to `anon_caps` on non-owner-scoped collections (no escalation). `/query` and `/query/explain` reject user tokens with 403 QUERY_USER_DENIED; `/mcp` rejects with 403 MCP_USER_DENIED. Stored RPCs accept user tokens when `anon_callable=true` and auto-bind `:user_id` from AuthCtx if declared and not in body; owner_field does NOT apply inside RPCs (S4 — RPC author owns the filter). Admin REST: `/t/<id>/admin/users` (service-only CRUD with cascade delete + revoke-sessions). MCP: 9 user/owner-field/self-register tools mirroring the REST surface. Admin UI: virtual sidebar entry `👤 _system_users` (password_hash column masked as `●●●●` in the rendered table). Audit rows are enriched with `auth_kind` and `auth_user_id` for every authenticated request, plus `email`/`ip_at_login` on login/register and `deleted_records`/`revoked_sessions` on admin user delete; auth bodies are NEVER persisted (S6). Daily janitor (`drust_session_janitor` binary) sweeps expired `_system_sessions` rows with 1d grace. Spec: [`docs/superpowers/specs/2026-05-09-drust-user-auth-design.md`](../docs/superpowers/specs/2026-05-09-drust-user-auth-design.md).
- **Per-tenant rmcp Streamable HTTP MCP endpoint at `/t/<tenant>/mcp`** serving the per-tenant tool set. One `StreamableHttpService<DrustMcpService>` per tenant, cached in `src/mcp/http_registry.rs`. MCP is **service-key-only** — anon bearers get `403 WRITE_DENIED`. Enforced in `src/tenant/mcp_dispatch.rs` before the dispatch; the route still runs through `bearer_auth_layer` so auth + rate-limit + audit all cover it. Canonical tool list: `src/mcp/handler.rs` (`#[tool]` annotations) — count is intentionally not pinned in this doc since it drifts. v1.7.3 added `set_anon_caps` (per-collection anon DML toggle, was UI-only) and `whoami` (returns tenant identity + both bearer tokens plaintext + REST/upload endpoint paths so models can hit the multipart upload route, which has no MCP tool by design).
- Admin UI is **two pages** (v1.5.1+): `/admin/tenants` (search-able list) and `/admin/tenants/{id}/<datatable>` (2-pane shell). The old stand-alone `/admin/tenants/{id}` detail page is gone — `GET /admin/tenants/{id}` 302-redirects to `/admin/tenants/{id}/_api_keys`. Six sidebar entries are virtual / always-shown (in order): `🔑 _api_keys` (anon · service · MCP setup, rendered by `tenant_api_keys.html`), `⚡ _rpc` (stored functions, added v1.6), `🔒 _system_files` (storage, rendered by `tenant_files_admin.html`), `👤 _system_users` (registered users, added v1.9), `🔐 _oauth_providers` (per-tenant Google/GitHub OAuth configs, added v1.12), `📋 _logs` (audit · request history, added v1.7.0) — then the real collections from `sqlite_master`. Canonical order is in `src/mgmt/templates/_collection_sidebar.html`. The MCP `claude mcp add-json` command lives inside `_api_keys` and copies via `window.location.origin` — no backend URL template needed.
- Admin UI (v1.2.0+): every page renders inside a viewport-fixed `.macwin` with container-scoped scroll. The 2-pane shell uses `grid-template-columns: var(--sidebar-w) minmax(0, 1fr)` so the right track can shrink below content min-content (long URLs, wide tables) instead of pushing cards past the sidebar boundary. `/admin/tenants/{id}/collections` 302-redirects to the first collection; empty tenants see a dedicated empty-state page. Breadcrumbs were removed from every admin page in v1.5.1 — the topbar `path` (`~/tenants/{id}/...`) carries clickable anchors per segment. The **LiveChonk** pixel-cat mascot (`_mascot.html`) is wired to any `<canvas class="pix" data-chonk=... data-size=...>` — 18 px in the topbar, 48 px on login, 96 px on empty states, 56 px on errors.
- **Admin audit UI** (v1.7.0+): two GET routes serving the existing audit JSONL files. `/admin/audit` (host) and `/admin/tenants/{id}/_logs` (per-tenant via sidebar virtual entry `📋 _logs`). Hard cap 50 000 entries per request — overflow truncated newest-first with UI footer warning.
- **Rate-limit + audit middleware are wired** into `bearer_auth_layer` (v0.1.0 Known issues closed in v1.1.1). Each tenant request produces one `audit-YYYY-MM-DD.jsonl` entry in `$DRUST_LOG_DIR`; denials get `error_code: HTTP_<status>`.
- **CORS** (v1.5.1+): tenant routes (`/t/<tenant>/...`) carry a `tower_http::cors::CorsLayer` applied **outside** `bearer_auth_layer` so `OPTIONS` preflight short-circuits before auth (preflight by spec carries no token). Allow-list comes from `DRUST_CORS_ORIGINS`; empty/unset disables the layer. Each entry is either an exact origin (`https://app.example.com`) or a single-`*` pattern — e.g. `https://*.tzuchi.org` matches any subdomain (refuses the bare apex), `http://localhost:*` matches any dev port. Multi-`*` patterns are rejected at parse time. Wildcard semantics live in `tenant::origin_matches` with unit tests against suffix-injection (`https://tzuchi.org.attacker.com`) and hyphen-confusion (`https://attacker-tzuchi.org`). Real GET/POST/etc. still flow through `bearer_auth_layer` unchanged — CORS only intercepts preflight and appends `Access-Control-Allow-Origin` to responses. Mgmt UI routes (`/admin/*`) have no CORS layer; they're server-rendered.

## Storage integration (Garage client, v1.4.0+)

Optional. Activated by setting `GARAGE_S3_ENDPOINT` + friends in `.env`. When enabled, drust gains `/drust/admin/files` (list / upload / delete / reconcile) backed by a `_system_files` metadata collection in `meta.sqlite`.

- **Garage is an independent service** (see `tool/garage/CLAUDE.md`). drust speaks plain S3 to it via `object_store::aws::AmazonS3`. drust boots with Garage unreachable — the storage tab shows "not configured" / admin operations return 503, but the rest of drust (tenants, MCP, REST, auth) is unaffected.
- **Reads bypass drust.** Anonymous GETs hit Caddy `/public/*` and `/t-public/<tenant>/*`, which reverse-proxy to Garage `s3_web` (`127.0.0.1:47831`) with `Host: public.web.local` or `Host: tenant-<id>-pub.web.local` respectively. drust is only in the *write* path.
- **SQLite-first upload / S3-first delete.** Upload inserts the metadata row, puts to Garage, and compensates by deleting the row on S3 failure. Delete calls Garage first (idempotent on NotFound), then clears the row. Orphans from partial failures are surfaced by the `reconcile` page.
- **`_system_*` collections are drop-protected.** `storage::schema::is_protected_collection()` is consulted by the `drop_collection` MCP tool. Applies to both the admin-level `_system_files` (in `meta.sqlite`) and the per-tenant `_system_files` (in each `data.sqlite`).

## Per-tenant files (Y scope, v1.5.0+)

When a tenant is created, drust auto-provisions two Garage buckets — `tenant-<id>-pub` (website enabled) and `tenant-<id>-prv` (private) — and grants the drust-client key owner access to both. Rollback on partial failure is compensating; leftover local state after Garage errors is captured in `_trash_pending_revokes` / `_orphan_buckets` for the `reconcile` page to retry.

- **Per-tenant file REST** at `/drust/t/<id>/files` — POST multipart upload / GET list / GET one / DELETE one / POST `<key>/sign` (pre-signed URL) / GET `<key>/bytes` (private proxy). Service-key-only.
- **Three file MCP tools**: `list_files`, `delete_file`, `get_file_url`. MCP has no upload tool; the `instructions` field on each tenant's MCP server directs the LLM to the REST endpoint with a curl example.
- **Admin UI parity** at `/drust/admin/tenants/<id>/files` — same upload/list/delete/sign form as the host-level `/admin/files`, scoped to the tenant's own buckets.
- **Disk guard**: uploads refuse with 507 when `/var/lib/garage` has less than `DRUST_DISK_MIN_FREE_PCT` (default 20) percent free.

> [!IMPORTANT]
> Garage's `s3_web` endpoint routes by **Host header** (`<bucket>.web.local`). The Caddy `reverse_proxy` for `/public/*` MUST carry `header_up Host "public.web.local"` or every request returns `NoSuchBucket`. Same family of gotcha as the MCP `header_up Host "127.0.0.1:47826"` below.

## Invariants

> [!WARNING]
> **Bearer tokens are the sole authorization boundary for data-plane access.** If a token leaks, it grants full read + structured write on the bound tenant until revoked. Never share tokens across tenants; never commit `.env`.

> [!CAUTION]
> **`header_up Host "127.0.0.1:47826"` is mandatory on the Caddy block** for the MCP sub-route `/drust/t/<tenant>/mcp`. rmcp's DNS-rebinding guard rejects non-loopback Hosts with a 403/421 that looks like a WAF.

> [!IMPORTANT]
> The SQL authorizer in `src/query/authorizer.rs` is the cross-tenant isolation guarantee at the SQL layer. If you loosen it (e.g. add new `AuthAction` allow arms), re-prove: (a) ATTACH stays denied, (b) sqlite_master reads stay denied, (c) all write actions stay denied on read connections.

> [!IMPORTANT]
> **User tokens (`drust_user_*`) cannot use `/query`, `/query/explain`, or `/mcp`** — drust does not rewrite user-supplied SQL, so row-level `owner_field` filters cannot be enforced on those surfaces. Reject with 403 QUERY_USER_DENIED / MCP_USER_DENIED. For per-user SELECTs against owner-scoped data, expose a stored RPC with `:user_id` (auto-bound from `AuthCtx`) — or use `/search` (v1.10.0+), which drust builds the SQL for so `owner_field` enforcement is guaranteed. For testing, use a service token. Documented in [`docs/superpowers/specs/2026-05-09-drust-user-auth-design.md`](../docs/superpowers/specs/2026-05-09-drust-user-auth-design.md) §1.

> [!IMPORTANT]
> **User tokens are allowed on `/search` but not `/query`.** Both endpoints accept structured input from the user, but `/query` accepts arbitrary SELECT SQL that drust does not rewrite — so `owner_field` cannot be enforced. `/search` accepts only a Filter AST (`src/query/vector_filter.rs`) that drust compiles to SQL with `?` binds, so the `owner_field` row-level filter is always enforced. If you add another endpoint that accepts user-supplied input that lands in SQL, decide explicitly which camp it belongs in.
